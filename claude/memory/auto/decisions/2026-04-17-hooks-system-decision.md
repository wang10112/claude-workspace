# 采用完整 Hooks 系统而非简单提醒

**日期：** 2026-04-17
**决策者：** 用户 + Claude
**标签：** #architecture #hooks #decision

## 背景

项目需要在特定事件（如文件写入、Git 提交）时自动执行检查和操作。

## 选项对比

### 方案 1：简单的 Markdown 提醒
- 在 hooks/ 目录放置 Markdown 文件
- 描述应该做什么
- 依赖人工执行

**优点：**
- 简单易懂
- 无需编写代码
- 维护成本低

**缺点：**
- 无法自动执行
- 容易被忽略
- 无法集成到工作流

### 方案 2：完整的 Hooks 系统
- JSON 配置文件定义 hooks
- Node.js 脚本实现逻辑
- 支持 JSON 输出控制行为

**优点：**
- 可以实际执行操作
- JSON 输出可以 block/allow
- 支持条件过滤（if 字段）
- 可以集成到 CI/CD

**缺点：**
- 需要编写脚本
- 配置稍复杂
- 学习曲线

## 决策

**选择方案 2：完整的 Hooks 系统**

## 理由

1. **自动化优先** - 用户偏好长期主义，提前规划完整方案
2. **功能强大** - 可以实际阻止危险操作（如提交硬编码密钥）
3. **可扩展** - 未来可以添加更多 hooks
4. **借鉴 ECC** - everything-claude-code 使用完整 hooks 系统，证明有效

## 实施方案

1. 创建 Hook 基类（hook-base.js）
2. 实现 3 个核心 hooks：
   - quality-gate.js - 代码质量检查
   - security-scan.js - 安全扫描
   - memory-capture.js - 记忆捕获
3. 提供 JSON 配置示例
4. 编写完整文档

## 影响

**正面：**
- 自动化质量检查
- 防止安全问题
- 自动构建知识库

**负面：**
- 初期开发时间增加（~90 分钟）
- 需要维护脚本

## 验证

- ✅ Hook 脚本可以正常执行
- ✅ JSON 输出控制有效
- ✅ 优雅失败机制工作正常

## 经验教训

- 长期主义的决策在后期会带来回报
- 完整的自动化系统比简单提醒更有价值
- 借鉴成熟项目（ECC）可以避免走弯路
