# 全局规则

## 语言规则

- AI 关键入口和机器面向的工作流文件使用英文，以提升系统行为
- 用户面向的说明文档优先使用中文，除非英文对 AI 系统运行必要
- 能力示例可以针对中文正式写作，但仓库结构保持通用性
- 文件和目录名使用小写字母和连字符，除非工具要求其他约定

## 架构规则

### 核心能力层 (claude/)
- 存储所有工作流、技能、角色、playbooks、模板、记忆和配置
- 是能力定义的唯一真实来源
- 包含：skills, workflows, memory, agents, playbooks, templates, commands, hooks, mcp, settings

### 文档层 (docs/)
- 存储架构和设置指南
- 存储扩展和维护说明
- 面向人类阅读

### 示例层 (examples/)
- 存储实际使用示例
- 存储学习路径和迁移映射

### 输出层 (output/)
- 存储最终交付物
- 按任务或项目组织

### 临时层 (tmp/)
- 存储临时参考资料
- 不作为策略或长期记忆

### 第三方依赖 (vendor/)
- 存储外部依赖（如 Anthropic 官方技能库）
- 使用 git submodule 管理，便于更新

## 维护规则

- 所有能力定义统一在 `claude/` 目录维护
- 持久化的工作区记忆存储在 `claude/memory/`，不要分散到临时文件
- 保持单一真源原则，避免重复定义

## 编辑器集成规则

- `.vscode/` 仅用于编辑器级集成
- MCP 定义应该对 VS Code Agent 模式有用
- Claude 特定的运行行为属于 `claude/` 目录

## 版本范围规则

- 保持第一版本最小化和可读性
- 优先支持规划、写作、审查和研究能力
- 避免构建完整的插件平台、安装器或巨大的命令目录
- 将成熟的能力模块吸收到工作区，而不是保持为独立的侧边工作区
- 当遗留工作区仍然有用时，将其视为参考源而不是能力的活跃主场

## 技能优先原则

- 技能是主要的工作流单元
- 命令是便利入口，不是能力本身
- 钩子提供流程提醒，不是能力本身
- 角色定义"谁"执行任务，技能定义"如何"执行

## 记忆持久化原则

- 长期记忆存储在 `claude/memory/` 的 Markdown 文件中
- 记忆文件可以被人类审阅和维护
- 记忆文件按模块组织（workspace, user-preferences, formal-writing, planning 等）
- 在执行任务前查阅相关记忆

## 扩展原则

- 新增能力直接在 `claude/` 对应目录创建
- 更新相关文档
- 添加示例和测试

详细扩展指南参考 `docs/EXTENDING-CAPABILITIES.md`
